# Phase 3 路线推荐模式与工具卡片设计
## 背景

当前游客端已经完成了以下几类能力：

- 左侧抽屉式历史会话入口
- GPT 风格的底部输入区
- `+` 菜单中的拍照上传入口
- “普通对话 / 路线推荐”两种 composer mode

但路线推荐模式目前还有两个明显问题：

1. 切换到路线推荐模式后，如果 session 已有兴趣标签，前端会立即触发推荐生成，而不是等待用户确认。
2. 路线推荐 UI 目前通过常驻的 `InterestTagPanel` 挂在聊天区正文顶部，挤占聊天区域高度，也会破坏消息区滚动体验。

这和目标中的 GPT 网页式体验不一致。用户希望：

- 模式切换后只出现一个清晰的蓝色模式标识
- 快捷问题跟着模式变化
- 聊天区和输入区整体形态不大改
- 路线推荐结果进入聊天流，而不是作为顶部工具面板常驻

同时，用户已经明确提出一个长期方向：路线推荐、拍照识别等能力未来最好能统一抽象成 `tools`。

本设计的目标是在不重写现有聊天主链和后端推荐接口的前提下，把路线推荐模式改造成 GPT 风格的“模式条 + 工具结果入对话流”结构，并为后续 `tools` 抽象预留稳定边界。

## 目标

- 去掉“切换到路线模式即自动生成推荐”的副作用
- 将路线模式的交互重心收回输入区上方，而不是聊天正文顶部
- 保持聊天区消息滚动和底部输入区布局稳定
- 让路线推荐结果作为一条结构化卡片消息进入聊天流
- 让路线推荐成为第一个 `tool-shaped` 的前端能力，为后续拍照识别等能力统一抽象提供样板

## 非目标

- 本轮不引入真正的 HTML 富文本聊天框，不支持任意富文本编辑
- 本轮不把整个聊天协议升级成完整 agent/tool-calling 框架
- 本轮不新增复杂后端编排服务、Redis 或新的工具运行时
- 本轮不重做现有推荐接口和推荐算法，只重组前端交互和消息承载方式
- 本轮不同时改造拍照识别为工具卡片消息，只为其后续抽象预留结构

## 方案决策

### 采用方案

采用“输入区模式条 + 手动生成 + 结构化工具卡片消息”的方案。

- 路线模式只改变输入区附近的 UI，不再在聊天正文顶部常驻大面板
- 切换到路线模式后，只出现蓝色模式条和对应的快捷提示，不自动请求推荐
- 兴趣标签选择区放在输入框上方，作为蓝色模式条展开后的内容
- 生成路线必须由用户手动触发
- 推荐结果进入聊天流，显示为一条路线推荐卡片消息
- 生成后继续停留在路线模式，直到用户手动切回普通对话

### 不采用的方案

- 不采用“最小修补”方案
  - 只去掉自动生成，但继续保留顶部 `InterestTagPanel`
  - 原因：虽然能修 bug，但聊天区仍被工具面板挤占，不符合 GPT 风格目标

- 不采用“通用富文本聊天框”方案
  - 原因：当前需求核心不是让用户输入富文本，而是让聊天消息能承载结构化工具结果
  - 通用富文本编辑器会显著增加实现复杂度和渲染风险

- 不采用“独立路线规划工作台/弹层”方案
  - 原因：会把当前单聊天流体验拆散，偏离用户明确提出的 GPT 网页式交互

## 总体交互设计

### 1. 路线模式入口

- 用户通过 `+` 菜单切换到路线推荐模式
- 切换成功后：
  - 输入区上方出现蓝色模式条
  - composer 的快捷问题切换为路线相关问题
  - 聊天区不自动新增推荐内容
  - 不触发推荐接口请求

蓝色模式条承担两个职责：

- 明确告诉用户当前在“路线推荐模式”
- 作为兴趣标签与“生成路线”按钮的展开入口

### 2. 路线模式展开区

展开区位于输入框上方，与 composer 绑定，而不是进入聊天正文。

展开区包含：

- 兴趣标签组
- 当前已选标签状态
- “生成路线”按钮
- 推荐失败时的轻量错误提示

交互规则：

- 用户可先切换到路线模式，再慢慢选择标签
- 未选标签时点击“生成路线”，前端不给空请求，只显示轻提示
- 选择/取消标签仍然同步保存到当前 session
- 切换模式本身不等于生成推荐

### 3. 路线推荐结果的呈现

推荐结果不再挂在聊天区顶部，也不再作为常驻工具面板刷新。

新的表现方式：

- 用户手动点击“生成路线”
- 前端调用现有推荐接口
- 成功后向消息流插入一条新的 assistant/tool result 消息
- 该消息渲染为“路线推荐卡片”

路线推荐卡片包含：

- 路线标题
- 简介
- 亮点标签
- 本次应用的兴趣标签
- 可点击的追问按钮

追问按钮的行为：

- 点击后发送一条普通文本问题
- 但保留 `meta=路线推荐`
- 仍然走现有聊天主链

### 4. 路线模式的驻留规则

生成完一条路线后：

- 默认继续停留在路线模式
- 不自动退回普通对话
- 用户只有手动切换模式时才退出路线模式

这样做更接近 GPT 的模式开关心智模型，也能支持用户连续生成不同路线方案。

## 消息结构设计

### 现状问题

当前 `ChatMessage` 主要承载文本正文、来源和追问标记：

- `content`
- `sources`
- `replyKind`
- `needsFollowup`

这套结构适合普通问答，但不适合承载路线推荐卡片这类结构化结果。

### 新结构方向

本轮不引入 HTML 富文本，而是把消息扩展成“文本 + 结构化块”。

建议在前端消息模型中增加一个可选的结构化结果字段：

- `toolResult`
  - `toolName`
  - `title`
  - `intro`
  - `highlights`
  - `appliedInterestTags`
  - `actions`

对路线推荐来说：

- `toolName = 'route_recommendation'`
- `actions` 表示卡片中的快捷追问按钮

这样有几个好处：

- 普通文本消息继续使用 `content`
- 工具结果可以按类型渲染成稳定卡片
- 不需要把推荐结果拼成大段字符串
- 为后续拍照识别、票务查询、演出时间查询等能力提供同一种消息承载方式

### 是否需要通用富文本

本轮结论是不需要。

原因：

- 路线推荐需求本质上是“结构化展示”，不是“任意富文本编辑”
- 使用结构化消息块比直接渲染 HTML 更安全、更可控
- 更适合后续统一抽象成 `tools`

## 前端组件边界

### `ChatComposer`

新增或承接以下职责：

- 显示蓝色路线模式条
- 展开/收起路线模式面板
- 展示兴趣标签与“生成路线”按钮
- 维持普通对话与路线模式的切换
- 保持拍照上传入口与录音/发送按钮不受影响

不负责：

- 展示路线推荐结果
- 直接渲染聊天消息中的卡片内容

### `InterestTagPanel`

不再作为聊天区顶部常驻大面板使用。

建议直接沿用现有 `InterestTagPanel` 并重构为更轻量的路线模式子面板，本轮不额外新增 `RouteModePanel`。

它的职责变为：

- 只服务输入区上方的路线模式展开区
- 管理兴趣标签展示
- 触发“生成路线”
- 显示轻量错误状态

### `ChatTranscript`

继续负责消息流和滚动，同时新增对“结构化工具结果消息”的渲染能力。

建议：

- 在消息渲染分支中识别 `toolResult`
- 将路线推荐消息交给独立卡片组件处理

### 新增卡片组件

建议增加：

- `ToolResultCard`
- `RouteRecommendationCard`

其中：

- `ToolResultCard` 作为通用封装层
- `RouteRecommendationCard` 处理路线标题、亮点、标签和追问按钮

这样可以让未来其他工具结果复用同样的框架。

## 与现有后端接口的关系

本轮尽量不改现有推荐接口和推荐算法。

### 保持不变的部分

- 推荐接口仍复用当前 `fetchVisitorRecommendations`
- 兴趣标签仍然保存到 session
- 推荐结果数据结构继续使用：
  - `routeTitle`
  - `intro`
  - `highlights`
  - `suggestedQuestions`
  - `appliedInterestTags`

### 本轮前端主改造

重点是前端状态和呈现层重组：

- 切模式时不自动请求推荐
- 推荐结果插入聊天流
- 路线面板从聊天正文顶部移回输入区上方

### 是否需要改后端消息 schema

优先不改。

本轮建议：

- 路线推荐消息先由前端本地组装为一条结构化 assistant 消息
- 不要求后端立即输出统一的 `tool_result` 事件

原因：

- 路线推荐是前端主动发起的独立 HTTP 请求
- 结果已经足够结构化
- 前端先把“工具卡片消息”这层抽象稳定下来，再考虑是否推广到后端统一事件协议

## 面向未来的 `tools` 抽象

本轮不正式引入完整 `tools framework`，但要做成 `tool-shaped`。

建议的长期方向：

- 每个工具结果都有统一元信息：
  - `toolName`
  - `title`
  - `inputSummary`
  - `resultPayload`
  - `actions`

路线推荐作为第一个样板工具：

- `toolName = route_recommendation`
- `inputSummary = 已选兴趣标签`
- `resultPayload = 路线推荐内容`
- `actions = 建议追问`

后续可自然扩展到：

- `spot_photo_recognition`
- `schedule_lookup`
- `ticket_info`
- `nearby_service`

这样前端最终会形成“聊天文本消息 + 工具结果消息”的双轨模型，而不需要演变成难以维护的富文本大杂烩。

## 错误处理

### 1. 未选标签直接生成

- 不发送推荐请求
- 在路线模式面板中提示“先选择至少一个兴趣标签”
- 不污染聊天消息流

### 2. 推荐接口失败

- 错误显示在路线模式面板
- 不插入失败卡片消息
- 用户可直接重新点击“生成路线”

### 3. 连续生成多条路线

- 每次成功生成都作为一条新的卡片消息插入聊天流
- 不覆盖旧路线
- 保留推荐历史，便于对比

### 4. 卡片追问失败

- 按普通聊天发送失败处理
- 保持卡片内容不丢失

## 测试与验收口径

### 前端行为验收

- 切换到路线模式后，不再自动生成路线
- 聊天消息区仍可正常滚动
- 输入区始终保持 GPT 风格的底部停靠结构
- 蓝色模式条出现且可展开/收起
- 快捷问题能随模式切换
- 标签选择和生成按钮位于输入区上方，而不是聊天正文顶部

### 路线推荐结果验收

- 点击“生成路线”后，推荐结果以卡片消息形式进入聊天流
- 卡片中可展示标题、简介、亮点、已选标签和追问按钮
- 点击追问按钮后，继续走现有聊天链路
- 生成完一条路线后，仍停留在路线模式

### 可扩展性验收

- 路线推荐卡片的消息结构不是临时字符串拼接
- 后续新增第二种工具结果时，可以复用同一类消息渲染模型

## 实现顺序建议

1. 去掉 `switchComposerMode()` 中切到路线模式时的自动推荐请求
2. 将顶部 `InterestTagPanel` 改为输入区上方的轻量路线模式面板
3. 扩展 `ChatMessage` 为可承载结构化工具结果
4. 在 `ChatTranscript` 中加入路线推荐卡片渲染
5. 将推荐接口结果插入聊天流
6. 补充测试并打磨视觉样式

## 结论

本轮路线推荐不会升级为通用富文本聊天框，而是升级为：

- GPT 风格的输入区模式条
- 手动触发的路线工具
- 进入聊天流的结构化工具卡片消息

这是一个兼顾当前交互质量与后续系统演进的中间方案：

- 先修正现有交互 bug
- 再建立稳定的“tool-shaped” 消息结构
- 为后续真正抽象成统一 `tools` 体系提供清晰落点
