# 景区导览 AI 数字人系统总体设计说明

## 1. 项目背景与目标

本项目面向景区导览场景，构建一套集文字问答、语音交互、Live2D 数字人展示、知识库问答、多模态识图、后台管理与数据分析于一体的 AI 数字人系统。

系统目标不是“能聊天”，而是让游客获得接近真人导游的体验：能问、能看、能听、能推荐、能追问，管理员也能通过后台持续配置数字人、补充知识、观察服务效果并形成闭环优化。

## 2. 需求分析

系统核心需求分为四类：

1. 游客端实时交互
   - 支持文本输入与语音输入
   - 支持 Live2D 数字人表情、动作与口型同步
   - 支持历史会话回看、路线推荐和拍照识图

2. 知识问答能力
   - 支持基于景区资料的 RAG 问答
   - 支持澄清追问与人性化改写
   - 支持知识缺口记录与后续补录

3. 管理后台能力
   - 支持数字人配置、音色管理、知识库管理、会话记录、数据大屏与感受度报告

4. 可部署与可验收
   - 支持本机 GPU 环境、测试服务器原生部署与 Docker 部署
   - 支持性能 trace、延迟分析和阶段性验收

## 3. 总体架构

系统采用“前端展示层 + FastAPI 服务编排层 + AI 与数据层”的三层架构。

```text
游客端 Vue3 / Live2D            管理后台 Vue3 / Ant Design
        │                                  │
        ├──────────── HTTP / WebSocket ────┤
        │                                  │
               FastAPI 服务编排层
   ASR / RAG / LLM / TTS / Vision / Trace / Admin API
        │
        ├──────── PostgreSQL / SQLite
        ├──────── ChromaDB / 本地知识库
        ├──────── DashScope / Qwen
        ├──────── faster-whisper / CosyVoice2
        └──────── 本地模型与上传资源
```

主数据流为：

```text
文本 / 语音 / 图片输入
  → ASR 或 Vision
  → RAG / LLM
  → 人性化改写与澄清判断
  → TTS 合成与情绪联动
  → Live2D 表情 / 动作 / 口型同步
```

## 4. 游客端设计

游客端采用单页导览台形态，围绕“对话 + 数字人 + 多模态输入”组织。

### 4.1 核心能力

- 文本对话：直接输入问题并获得流式回复
- 语音对话：录音后经 ASR 转文本，再进入同一聊天链路
- 会话历史：侧边栏展示历史会话，支持切换回看
- 路线推荐：基于兴趣标签生成推荐路线与追问建议
- 拍照识图：上传图片后由视觉模型识别景点，再转入现有对话链

### 4.2 数字人表现

- Live2D 舞台展示数字人形象
- 情绪灯用于提示当前情绪与状态
- 思考态用于用户提问后的等待反馈
- 发声态与口型同步驱动嘴型参数
- 闲置态用于无对话时的自然待机

### 4.3 多模态接入方式

图片能力采用“HTTP 识图后转入聊天”的方式实现：

1. 前端上传图片
2. 后端调用 Qwen-VL-Max 识别景点
3. 后端生成适合继续提问的自然语言问题
4. 前端把该问题作为普通文本消息送入 WebSocket 对话主链

这种设计能保证图片能力、RAG、TTS、口型同步与会话持久化继续复用同一条主链，避免协议重构。

## 5. 管理后台设计

管理后台用于支撑数字人运营与持续迭代，主要包括以下模块：

- 数字人配置：Live2D 模型、显示参数、音色、系统人设、TTS 参考音频等
- 知识库管理：文档上传、解析、切片、入库、删除与状态查看
- 音色管理：参考音频上传、校验、试听与下拉选择
- 会话记录：将 messages 真正可视化，便于回溯访谈过程
- 知识缺口：统计无法被现有 RAG 回答的问题，并支持人工补答后回流知识库
- 数据大屏：服务量、热门问题、满意度趋势、关键词云、在线人数
- 感受度报告：按日期范围生成情绪与反馈分析报告

后台形成的闭环是：

```text
配置数字人 → 运行服务 → 观察会话与报表 → 发现知识缺口 → 人工补充 → 再入知识库
```

## 6. AI 核心链路设计

### 6.1 ASR

- 使用 `faster-whisper`
- 支持本地语音输入识别
- 语音结束后直接转写为文本进入同一聊天链路

### 6.2 RAG

RAG 链路采用：

```text
用户问题 → bge-m3 向量检索 → bge-reranker-v2-m3 重排 → Qwen 生成
```

当前实现不直接拼贴知识库原文，而是先做导览式重写，再决定是否需要澄清追问。这样能减少“像在念资料”的机器感。

### 6.3 人性化改写与澄清追问

系统会对检索结果进行结构化整理：

- 先回答游客真正关心的核心结论
- 只保留与问题直接相关的少量补充信息
- 对边界不清的问题先答后问
- 对信息不足的问题生成澄清追问并持久化状态

### 6.4 TTS 与情绪联动

当前主 TTS 方案为：

```text
CosyVoice2-0.5B + inference_instruct2
```

系统将情绪标签映射为语音指令，让声音风格与回答内容联动；同时保留流式输出、分句合成、参考音频缓存与 trace 记录，以支持更低首包延迟和更容易排查的播放链路。

### 6.5 口型与动作协调

口型同步以 CosyVoice2 的结构化 timing 为优先；当 timing 不完整时退回到音频包络近似方案。

动作层面采用“thinking → speaking → cooldown → idle”的阶段机，让数字人在等待、发声和回落时都有明确表现，不只是随机切换 idle 动画。

## 7. 数据与接口设计

### 7.1 公开接口

#### 游客端 WebSocket 对话通道

统一为单会话实时对话通道，支持：

- `text`
- `audio_chunk`
- `audio_end`
- `text_delta`
- `asr_result`
- `emotion`
- `avatar_phase`
- `tts_audio_chunk`
- `tts_viseme_chunk`
- `audio_done`
- `done`

#### 游客端 HTTP 接口

- 会话创建与列表
- 会话消息读取
- 会话兴趣标签更新
- 路线推荐
- 图片上传与识图

#### 管理端 HTTP 接口

- 登录认证
- 知识库管理
- 数字人配置
- 音色资源管理
- 会话记录
- 知识缺口管理
- 数据大屏
- 感受度报告

### 7.2 核心数据表

- `sessions`：游客会话与兴趣标签
- `messages`：对话消息与情绪、延迟信息
- `avatar_config`：数字人模型、音色、人设与 TTS 配置
- `conversation_states`：澄清追问状态
- `voice_profiles`：参考音频与音色资源
- `knowledge_docs`：知识库文档与处理状态
- `knowledge_gaps`：无法回答问题的统计与人工补答入口
- `daily_emotion_reports`：每日情绪报告

## 8. 部署与运行环境

项目同时支持本机开发、测试服务器原生部署和 Docker 部署。

### 8.1 本机开发

- 前端：Vite + Vue3
- 后端：FastAPI + uvicorn
- 数据库：PostgreSQL
- 模型：本地 GPU 环境加载 Whisper、CosyVoice2、Embedding 模型

### 8.2 测试服务器

- 支持原生部署和 Docker Compose 部署
- 支持 V100、A100 等 GPU 测试环境
- 也兼容当前本机 4060 开发机的联调模式

### 8.3 运行资源

- 文档、知识库、图片上传、音色与 TTS 参考音频均存放于 `backend/storage`
- 服务器侧可使用独立的模型目录与数据目录

## 9. 性能与稳定性设计

系统对关键耗时做了 trace：

- ASR 模型加载与转写耗时
- RAG 检索、重排、生成耗时
- TTS chunk 级首包、间隔与发送耗时
- 前端播放缓冲、underrun 与阶段切换耗时

当前优化策略包括：

- 流式文本输出
- 分句 TTS 合成
- 本地知识库检索
- 参考音频缓存
- 前端 jitter buffer
- 后端 trace 异步落盘

总体上，系统已经具备可演示链路，但 TTS 卡顿、口型量化验收和部分阶段性能仍在继续打磨。
